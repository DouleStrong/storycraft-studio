# StoryCraft Studio 正文质量优化经验记录

日期：2026-04-10

## 这轮遇到的真实问题

### 1. Writer 能写通，但容易写成“安全模板腔”

最明显的症状有三类：

- 段落都在做同一种事：铺气氛、讲心理、收悬念，缺少明确分工
- 情绪表达偏抽象，喜欢直接总结人物状态，而不是用动作、停顿、对白去交付
- 高频落入通用套话，例如“往事如潮水”“未来的阴影”“某种说不清的感觉”

结论：

- 只要求 `narrative_blocks: list[str]`，会让模型更容易产出“看起来像正文、实际上缺少戏剧骨架”的文本
- 如果不把“每段为什么存在”说清楚，模型会自动选择最安全的写法

### 2. Story Bible 存在数据库里，不代表它真的影响了正文

之前的链路里，`addressing_rules`、`timeline_rules` 虽然已经有字段，但没有真正进入 Writer 的上下文。

后果：

- 角色称呼容易漂移
- 时间线约束只在事后审校时才被发现
- Story Bible 变成“可编辑资料”，而不是“生成时真正生效的规则中枢”

结论：

- 设定必须进入 prompt，而不是只存在后端模型里

### 3. Reviewer 默认整章回填，会把 Writer 的锋利度磨平

之前的链路中，只要 Reviewer 返回了 `revised_narrative_blocks`，落库阶段就优先采用它。

这会带来一个副作用：

- 一致性更高
- 但 Writer 的句法节奏、局部锋利表达、人物差异感会被 Reviewer 的“统一口径”收平

结论：

- “审校成功”不应该等于“整章重写后再落库”
- 对轻微问题，更合理的策略是保留 Writer 原稿，只补充说明或做局部修补

### 4. 想验证正文质量时，不应该被图像链路绑架

在真实 smoke 里，我们多次遇到：

- `outline` 和 `draft` 已经跑通
- 但 image provider 限流导致整条 smoke 失败

这会造成误判：

- 容易把“图像阶段失败”误认为“正文阶段没有改进”

结论：

- 评估正文质量时，应该能单独停在 `draft` 或 `scenes` 阶段
- 以后建议给 smoke 工具加 `stop_after=draft|scenes|illustrations`

## 这轮采取的改法

### 1. 给 Writer 增加真正影响文本质感的硬约束

这轮不是简单提高温度，而是直接把下面这些要求写进 Writer prompt：

- 每段必须承担明确的戏剧功能
- 至少一段通过具体动作推进
- 至少一段通过对白、潜台词或即时反应暴露关系变化
- 至少一段交付新信息、误导纠正或决策代价
- 禁止常见套话
- 项目为中文叙事时，不要无故切英文人名或拼音称呼

结论：

- 对正文质量最有效的不是“更大而空泛的风格描述”，而是“明确写作禁令 + 具体动作要求 + 段落功能要求”

### 2. 把作者已确认的文本样本作为 style memory 喂回模型

我们发现最值钱的上下文，不是整本项目摘要，而是：

- 已锁定正文块
- 用户手改过的正文块
- 最近几章已经成型的具体段落样本

这些样本比“tone=克制、电影感”更能告诉模型：

- 叙事距离该有多近
- 句子该有多短
- 人物压强应该怎么落在动作和对白上

结论：

- 下一轮如果继续做正文质量，优先扩大 style memory，而不是盲目堆更多 world notes

### 3. 给 Reviewer 增加显式 `apply_mode`

这轮新增了 `apply_mode`：

- `apply_revisions`
- `preserve_writer`

意义：

- Reviewer 可以明确表达“这稿可用，我建议保留 Writer 原文”
- 落库层不用再猜 Reviewer 是想全面接管，还是只想提出建议

结论：

- 生成链路里，Agent 间的“采纳策略”应该显式建模，不要只靠文案语义猜测

### 4. 需要一层轻量 quality flags，防止 Reviewer 误保留“坏味道正文”

真实抽样里我们看到，即使 prompt 已经加强，模型仍可能产出：

- 高频套话
- 空泛情绪总结
- 偏概述而不够贴身的句子

因此这轮增加了一层轻量 `quality_flags`：

- 不做复杂 NLP
- 只抓高频坏味道短语
- 把命中结果显式传给 Reviewer

这样做的价值是：

- 不依赖 Reviewer 自己“碰巧注意到”
- 可以在 `preserve_writer` 和 `apply_revisions` 之间增加一个更稳的决策依据

结论：

- 对长文本质量，完全相信 LLM 自检是不够的
- 简单的本地启发式规则，往往比继续堆 prompt 更稳

## 下次开发时建议直接沿用的做法

### 1. 先用 provider-level request test 卡 prompt，再改模型策略

这轮最有效的测试不是大而全的 e2e，而是：

- 直接检查发给 Writer/Reviewer 的 request body
- 验证关键规则有没有真的进入 prompt

原因：

- 这类问题的根因往往不是数据库或 API，而是“你以为模型看到了，但其实没看到”

### 2. 对正文质量问题，必须做真实样本抽检

只看单元测试不够，因为：

- 测试只能保证链路逻辑成立
- 不能保证文字读起来真的不平

建议流程：

1. 先跑最小 smoke 到 `draft`
2. 抽 1-2 段真实正文
3. 结合 prompt 和持久化策略一起诊断

### 3. 不要把“审校一致性”当成唯一目标

如果 Reviewer 的目标只剩“统一”，最终成品会：

- 更稳
- 但更平

另外，这轮真实抽样还说明了一点：

- 即使链路已经比之前更合理，`gpt-4o-mini` 在文学性、细节密度和句法变化上仍然容易保守

结论：

- 链路优化和模型档位是两个问题
- 链路可以把“错误方向”纠正过来，但不能把中档模型直接拉到出版级正文水准
- 如果目标是接近交付稿，下一轮应该优先尝试单独升级 `STORY_AGENT_WRITER_MODEL` 与 `STORY_AGENT_REVIEWER_MODEL`

下一轮建议把 Reviewer 角色进一步分成两种工作模式：

- `consistency_guard`: 只抓口吻、称呼、时间线、设定冲突
- `craft_editor`: 只在用户明确触发时，帮助做更强的文学润色

这样可以避免默认链路把所有稿子都磨成一个腔调。

## 建议的下一步

如果继续提升正文质量，优先顺序建议是：

1. 把章节规划升级为 beat 级，不再只给 `summary/goal/hook`
2. 给 WriterDraft 增加 block 级功能标签，而不只是 `list[str]`
3. 给 smoke 工具增加 `stop_after`，降低排查成本
4. 把 style memory 正式产品化，在前端让作者看到“这轮参考了哪些已确认文本”
